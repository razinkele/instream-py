# v0.49.0 Create Model CSV Export Format Fix — Design Spec

**Date:** 2026-04-26
**Branch base:** `master` at `60db5c0` (post-v0.48.0 merge)
**Release tag (target):** `v0.49.0`
**Status:** Approved at brainstorm. Single-file bug fix scoped tightly to one function + one round-trip regression test.

## Goal

Fix `app/modules/create_model_export.py::export_template_csvs` so the per-cell hydraulic CSVs it produces can be loaded by `salmopy.io.hydraulics_reader._parse_hydraulic_csv`. **The bug is purely format mismatch**, empirically demonstrated by `scripts/_probe_create_model_csv_format.py` which fails today with `ValueError: invalid literal for int() with base 10: 'flow'`. After this release, exporting a fixture from the Create Model UI and feeding it back into the simulation works end-to-end.

## The bug

**Current `export_template_csvs` output** (broken — `flow` as first row, cells as columns):
```
flow,C0001,C0002,C0003
0.5,0.5234,0.6311,...
1.0,0.5891,...
...
```

**Format expected by `_parse_hydraulic_csv`** (canonical, e.g. `tests/fixtures/example_baltic/Nemunas-Depths.csv`):
```
; Depths for Baltic example — Nemunas
; depth_base/vel_base are real published means; per-flow scaling is synthetic
; CELL DEPTHS IN METERS
10,Number of flows in table,,,,,,,,,,,,,,,,,,
,1.0,2.0,4.0,6.0,9.0,14.0,22.0,40.0,100.0,500.0
1,4.874167,5.200352,5.526536,...
2,5.924315,...
```

The two formats are **transposed** (loader expects cell rows × flow columns) AND the loader requires a comment + count line + flow-values line preamble (3 structural sections) that the export omits entirely.

## Architecture

Single-file fix in `app/modules/create_model_export.py`. Replace the two `pd.DataFrame(...).to_csv(path, index=False)` calls (one for Depths, one for Vels) with a manual CSV writer that:

1. Emits a 5-line `;`-prefixed comment header (auto-documenting the synthetic template + a positional-contract warning that the row order must match the shapefile cell order).
2. Emits the count line: `<n_flows>,Number of flows in table` followed by `n_flows - 1` trailing commas to match canonical width.
3. Emits the flow-values line: empty first column, then comma-separated flow values.
4. Emits one row per cell: `<cell_id>,<val_at_flow_1>,<val_at_flow_2>,...`.

The TimeSeriesInputs.csv writer (using a different format that's already loader-compatible) stays unchanged. The `export_zip` caller (line 279, the call site inside `export_zip`'s body) is unchanged. The depth/velocity formulas are unchanged — they remain synthetic placeholders for the user to overwrite before calibration.

## Components

### Sub-task 1 — Refactor `export_template_csvs` (commit 1)

**File:** `app/modules/create_model_export.py`

**Add a private helper** at the same module level:

```python
def _write_hydraulic_csv(
    path: Path,
    *,
    reach_name: str,
    kind: str,             # "Depths" or "Vels"
    units: str,            # "METERS" or "M/S"
    flows: list[float],
    cell_ids: list[str],
    values: list[list[float]],   # values[i][j] = value at cell i, flow j
) -> None:
    """Write a per-cell hydraulic CSV in the format expected by
    salmopy.io.hydraulics_reader._parse_hydraulic_csv.

    Format (5 comment lines + count + flow-values + n_cells data rows):
      ; <kind> for <reach_name> — generated by Create Model export
      ; Synthetic template values — replace with measured hydraulics before calibration
      ; IMPORTANT: row order must match shapefile cell order within this reach.
      ; The simulation indexes hydraulic values positionally against the shapefile.
      ; CELL <KIND> IN <UNITS>
      <n_flows>,Number of flows in table,,,...    (n_flows - 1 trailing commas)
      ,<flow_1>,<flow_2>,...                       (empty first column)
      <cell_id_0>,<value_0_0>,<value_0_1>,...
      <cell_id_1>,...
    """
    n_flows = len(flows)
    if n_flows == 0:
        raise ValueError(f"{path}: flows must not be empty")
    if any(len(row) != n_flows for row in values):
        raise ValueError(
            f"{path}: each cell's value row must have exactly {n_flows} entries"
        )
    if len(values) != len(cell_ids):
        raise ValueError(
            f"{path}: cell_ids ({len(cell_ids)}) and values ({len(values)}) "
            "must have the same length"
        )
    lines: list[str] = []
    lines.append(f"; {kind} for {reach_name} — generated by Create Model export")
    lines.append("; Synthetic template values — replace with measured hydraulics before calibration")
    lines.append("; IMPORTANT: row order must match shapefile cell order within this reach.")
    lines.append("; The simulation indexes hydraulic values positionally against the shapefile.")
    lines.append(f"; CELL {kind.upper()} IN {units}")
    # Count line: trailing commas are ignored by the reader (it splits on `,`
    # and reads only the first field). Pad to n_flows-1 trailing commas as a
    # spreadsheet-friendly width hint; the canonical example_baltic file uses
    # a wider pad but the reader treats both identically.
    lines.append(f"{n_flows},Number of flows in table" + "," * (n_flows - 1))
    # Flow-values line: empty first column, then flows
    lines.append("," + ",".join(f"{f}" for f in flows))
    # Data rows: cell_id, val_at_flow_1, val_at_flow_2, ...
    for cid, row in zip(cell_ids, values):
        lines.append(f"{cid}," + ",".join(f"{v:.4f}" for v in row))
    # Force LF line endings (Path.write_text on Windows would write CRLF
    # via universal-newlines translation, diverging from the canonical
    # example_baltic format). The reader is tolerant of either, but byte-
    # stable output across platforms is the cleaner contract.
    path.write_bytes(("\n".join(lines) + "\n").encode("utf-8"))
```

**Modify `export_template_csvs`** to use the helper. Replace the existing `pd.DataFrame(depth_rows)...` and `pd.DataFrame(vel_rows)...` blocks with calls to `_write_hydraulic_csv`. The existing per-cell formulas stay unchanged. After the fix, the function body looks like:

```python
def export_template_csvs(
    reaches: dict,
    cells_gdf: gpd.GeoDataFrame,
    output_dir: Path,
    start_date: str,
) -> list[Path]:
    """Generate template CSV files (time-series, depths, velocities) per reach.

    Returns list of paths to created CSV files.
    """
    output_dir = Path(output_dir)
    output_dir.mkdir(parents=True, exist_ok=True)
    created: list[Path] = []
    dates_365 = _date_range_365(start_date)
    flows = TEMPLATE_FLOWS

    for rname in sorted(reaches.keys()):
        reach_cells = cells_gdf[cells_gdf["reach_name"] == rname]
        cell_ids = list(reach_cells["cell_id"])
        n_cells = len(cell_ids)
        if n_cells == 0:
            continue

        # --- TimeSeriesInputs (365 rows) — UNCHANGED ---
        ts_path = output_dir / f"{rname}-TimeSeriesInputs.csv"
        ts_df = pd.DataFrame(
            {
                "date": dates_365,
                "flow": [5.0] * 365,
                "temperature": [10.0] * 365,
                "turbidity": [1.0] * 365,
            }
        )
        ts_df.to_csv(ts_path, index=False)
        created.append(ts_path)

        # --- Depths (n_cells rows × n_flows columns, transposed from old format) ---
        depth_path = output_dir / f"{rname}-Depths.csv"
        depth_values: list[list[float]] = []
        for cid in cell_ids:
            cell_var = 0.8 + 0.4 * (hash(cid) % 100) / 100.0
            depth_values.append([
                round(
                    (0.3 + 0.7 * math.log(flow / 0.5 + 1) / math.log(1001)) * cell_var,
                    4,
                )
                for flow in flows
            ])
        _write_hydraulic_csv(
            depth_path,
            reach_name=rname,
            kind="Depths",
            units="METERS",
            flows=flows,
            cell_ids=cell_ids,
            values=depth_values,
        )
        created.append(depth_path)

        # --- Velocities ---
        vel_path = output_dir / f"{rname}-Vels.csv"
        vel_values: list[list[float]] = []
        for cid in cell_ids:
            cell_var = 0.8 + 0.4 * (hash(cid) % 100) / 100.0
            vel_values.append([
                round((0.1 + 0.5 * flow / 50) * cell_var, 4)
                for flow in flows
            ])
        _write_hydraulic_csv(
            vel_path,
            reach_name=rname,
            kind="Vels",
            units="M/S",
            flows=flows,
            cell_ids=cell_ids,
            values=vel_values,
        )
        created.append(vel_path)

    return created
```

The only behavioural difference vs. today is **the file format on disk**. The mathematical placeholder values are byte-identical (same hash-based per-cell variation, same log/linear formulas). Cell IDs are still `Cnnnn` strings from the GeoDataFrame.

```python
def test_export_template_csvs_round_trips_through_hydraulic_reader(tmp_path):
    """Round-trip: export 3-cell template → load via _parse_hydraulic_csv.

    This is the test that, when missing pre-v0.49, allowed the format
    mismatch (loader expects transposed matrix + comment+count+flow-values
    header; export emitted flow-as-row + cells-as-columns + no header)
    to ship undetected since v0.30.x. Locking it in.
    """
    # All imports done at module scope (see "Module-level additions" above).
    # The test body uses: np, gpd, pd, Polygon, export_template_csvs,
    # TEMPLATE_FLOWS, _parse_hydraulic_csv.

    cells = gpd.GeoDataFrame(
        {
            "cell_id": ["C0001", "C0002", "C0003"],
            "reach_name": ["R1", "R1", "R1"],
        },
        geometry=[Polygon([(0, 0), (1, 0), (1, 1), (0, 1)]) for _ in range(3)],
        crs="EPSG:4326",
    )

    paths = export_template_csvs(
        reaches={"R1": {}},
        cells_gdf=cells,
        output_dir=tmp_path,
        start_date="2026-04-26",
    )

    depths_path = tmp_path / "R1-Depths.csv"
    vels_path = tmp_path / "R1-Vels.csv"
    ts_path = tmp_path / "R1-TimeSeriesInputs.csv"
    assert depths_path in paths
    assert vels_path in paths
    assert ts_path in paths

    # Round-trip Depths
    flows, depths, cids = _parse_hydraulic_csv(depths_path, return_cell_ids=True)
    assert flows.shape == (10,)
    assert depths.shape == (3, 10)
    assert cids == ["C0001", "C0002", "C0003"]
    # Lock in the TEMPLATE_FLOWS contract — guards against accidental
    # changes to the flow grid that would silently break downstream callers.
    assert list(flows) == pytest.approx(TEMPLATE_FLOWS)
    # Sanity: the depth matrix must contain SOME variation. Guaranteed by
    # the formula `(0.3 + 0.7 * log(flow/0.5+1)/log(1001)) * cell_var`:
    # base(flow) is monotonic in flow, so depths[c, 0] != depths[c, -1] for
    # ANY cell c regardless of cell_var. This avoids relying on Python's
    # randomized string-hash for cell_var-driven inter-cell variation.
    assert not np.all(depths == depths[0, 0])

    # Round-trip Vels
    flows_v, vels, cids_v = _parse_hydraulic_csv(vels_path, return_cell_ids=True)
    assert flows_v.shape == (10,)
    assert vels.shape == (3, 10)
    assert cids_v == ["C0001", "C0002", "C0003"]
    assert list(flows_v) == pytest.approx(TEMPLATE_FLOWS)
    assert not np.all(vels == vels[0, 0])

    # Round-trip TimeSeriesInputs (pandas-loadable, separate format from
    # the hydraulic CSVs but worth a smoke check while we're here).
    ts = pd.read_csv(ts_path)
    assert len(ts) == 365
    assert set(ts.columns) >= {"date", "flow", "temperature", "turbidity"}
```

Four assertions matter most:
- `depths.shape == (3, 10)` — proves the matrix is transposed correctly (3 cells × 10 flows, not 10 × 3).
- `cids == ["C0001", "C0002", "C0003"]` — proves the loader's first-column extraction sees the cell IDs, not a stray `flow` literal.
- `list(flows) == pytest.approx(TEMPLATE_FLOWS)` — locks in the flow-grid contract; future changes to `TEMPLATE_FLOWS` must be deliberate.
- `not np.all(depths == depths[0, 0])` — proves the matrix has SOME variation across the 3×10 grid. Hash-collision-resistant: monotonic flow → `depths[0, 0] != depths[0, -1]` regardless of cell_var.

## Architecture / errors / testing

- **No simulation behavior change.** Existing fixtures (example_a, example_b, example_baltic, 4 WGBAST) are untouched. Their CSVs were never produced by `export_template_csvs` — they were authored by hand or by `_wire_wgbast_physical_configs.py`, both of which already use the canonical format. Only fixtures freshly exported via the Create Model UI are affected.
- **No new error paths in `export_template_csvs`.** The new `_write_hydraulic_csv` helper validates input shape (`n_flows > 0`, `len(values) == len(cell_ids)`, each `values[i]` has `n_flows` entries) and raises `ValueError`. Callers don't need to catch — the existing function has no `try/except`.

- **Hidden positional contract surfaced in the file's comment header.** The hydraulic CSVs and the shapefile must agree on cell row order: `model_init.py` indexes hydraulic values positionally (not by cell_id string). The new `_write_hydraulic_csv` helper writes a comment line in every output file that warns: `; IMPORTANT: row order must match shapefile cell order within this reach.` This pre-existing contract was previously undocumented; the fix makes it visible at every export.
- **One regression test covers the contract.** A round-trip test that exports + re-reads is the cleanest possible safety net for a format-of-output bug.
- **Probe script deleted.** `scripts/_probe_create_model_csv_format.py` was a throwaway investigation; the regression test supersedes it.

## Out of scope

- **Real (non-synthetic) depth/velocity formulas.** The function is `export_template_csvs` — synthetic placeholders are by design.
- **TimeSeriesInputs.csv format.** Already loader-compatible.
- **YAML / shapefile / ZIP export pipelines.** All working; this fix is scoped to the per-cell hydraulic CSV pair only.
