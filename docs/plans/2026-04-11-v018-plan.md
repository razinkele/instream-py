# v0.18.0 Implementation Plan — Calibration Trustworthiness

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:executing-plans` (inline) or `superpowers:subagent-driven-development`. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Restore clean full-suite CI by fixing the v0.17.0 test-order flake, then elevate the calibration test from a collapse-detector to a quantitative point-calibration by introducing a dedicated Baltic Atlantic salmon species config and tightening the SAR band from 2–18% to 3–12%.

**Architecture:** Three phases with a dependency order. Phase 1 diagnoses and fixes `test_marine_e2e.py::TestMarineLifecycleE2E::test_freshwater_still_works` — which passes in isolation and small subsets but fails in the 882-test full suite — via pytest bisection and module-state inspection. Phase 2 creates a new `configs/baltic_salmon.yaml` species config based on published Atlantic salmon bioenergetics (Hanson et al. 1997, Handeland et al. 2008) and migrates `tests/test_calibration_ices.py` to use it. Phase 3 tightens the SAR band and updates `docs/calibration-notes.md` with the species-mismatch disclaimer removed. Phase 4 is release hygiene.

**Tech Stack:** Python 3.13, pytest 9 (with `--lf` and `-p no:randomly`), numpy 2.x, pydantic 2.x, micromamba `shiny` environment, scite MCP for new Atlantic salmon parameter citations.

**Branch / worktree:** work inline on master (v0.17.0 already released as `00465fc` + tag `v0.17.0`).

**Risk budget:**

1. **Flake is nondeterministic, not order-dependent** (cycle-1 review finding). The test-infrastructure reviewer traced the leak to `src/instream/model_init.py:396`, where `MarineDomain(...)` is constructed **without** passing `self.rng`. This means `MarineDomain` falls into the `np.random.default_rng()` default-branch at `src/instream/marine/domain.py:138` and creates a fresh **OS-entropy-seeded Generator** every run. The test is flaky because different random draws at ocean-phase kill decisions produce different extinction outcomes — this has **nothing to do with test order**. Bisection would chase a ghost. **Mitigation**: Phase 1 is rewritten to thread `self.rng` into `MarineDomain` as the first action. Run the suite 3× to confirm determinism. No bisection unless the determinism fix does not fix the flake.

2. **Baltic salmon bioenergetics parameters**: the plan cites Hanson 1997 and Handeland 2008 as the sources. If scite retrieval shows these papers don't give the exact values we need, we fall back to Atlantic salmon values from existing Baltic-salmon literature (Kallio-Nyberg et al., Niemelä et al., already cited in v0.17.0 `docs/calibration-notes.md`). **Mitigation**: Phase 2 Task 2.1 does the literature search FIRST, before writing the config.

3. **SAR band tightening breaks the calibration test**: after moving to Baltic salmon bioenergetics AND tightening the band, the calibration run may miss the new band even though the pipeline is correct. **Mitigation**: Phase 3 has an explicit re-tune step analogous to v0.17.0 Phase 4 Task 4.3 staged tuning, with a 3-edit ceiling.

4. **`example_calibration.yaml` still references `Chinook-Spring`**: once the Baltic config exists, we must decide whether the calibration test uses the Baltic config directly OR keeps `example_calibration.yaml` but swaps its species block. Decision: **create a second config** `configs/example_calibration_baltic.yaml` that inherits the structure of `example_calibration.yaml` but imports the Baltic species. This preserves the v0.17.0 `example_calibration.yaml` for backward compatibility and regression testing.
