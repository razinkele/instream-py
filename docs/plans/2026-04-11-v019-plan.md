**Branch / worktree:** work inline on master (v0.18.0 already released as `8baa869` + tag `v0.18.0`).

**Risk budget:**

1. **Phase 1 flake sibling-state investigation is unbounded.** v0.18.0 Phase 1 established that `test_marine_e2e.py::test_freshwater_still_works` is a DETERMINISTIC test-order interaction (not an RNG flake — that was fixed in v0.18.0 by threading `self.rng` into `MarineDomain`). Two consecutive full-suite runs produced identical failure. The root cause is some upstream test polluting the marine cohort's final population. **Mitigation**: hard 45-minute time-box on Phase 1 pytest bisection. If root cause is identified and fixed, remove the `@pytest.mark.xfail` marker. If not identified in 45 min, keep the xfail marker and document the investigation attempt in CHANGELOG — no net change.

2. **`spawn_defense_area` semantic reconciliation is a breaking change**. NetLogo uses it as an **area in m²** subtracted from available spawning gravel per existing guarder (`InSALMO7.3_2023-07-06_ExampleA.nlogo:4200`). Python port uses it as a **Euclidean distance exclusion radius** in cm (`src/instream/modules/spawning.py:186`). Any existing config with `spawn_defense_area: 200000` expects one semantic and breaks under the other. **Mitigation**: Phase 3 renames the Python field to `spawn_defense_area_cm` (preserving the distance semantic) and adds a new `spawn_defense_area` field with the NetLogo area semantic. Both configs and the existing `example_calibration.yaml` / `example_calibration_baltic.yaml` are unchanged (they use `0` so neither semantic applies). This preserves backward compatibility.

3. **Baltic horizon extension may require re-calibration**. Extending the Baltic calibration from 5 to 7 years gives the cohort more time to die from attrition. If SAR drops below the 3% floor after the extension, the test will fail even though the ecological model is correct. **Mitigation**: if Phase 2 produces SAR < 3%, reduce seal hazard by 30% (0.010 → 0.007) as a single-edit re-tune. 1-edit ceiling.

4. **Phase 4 scite sweep creep**. All 86 Chinook-copied fields could theoretically benefit from Atlantic-salmon retrieval, but most are behavioural tuning knobs with near-zero effect on calibration. **Mitigation**: Phase 4 is hard-scoped to exactly **6 fields** with maximum scientific leverage: `spawn_fecund_mult`, `spawn_fecund_exp`, `redd_devel_A`, `redd_devel_B`, `redd_devel_C`, `energy_density`. The remaining 80 fields are explicitly deferred to "v0.20.0+ if ever" — not a known gap, not a roadmap item, just "if someone has time".

## Phase 1 — Resolve `test_freshwater_still_works` flake (time-boxed 45 min)

**Context** (critical — read before bisecting): v0.18.0 Phase 1 established two facts that shape Phase 1's approach:

1. **It's not an RNG flake.** `MarineDomain` was constructed without `self.rng` in v0.17.0, producing OS-entropy seeded kill draws. v0.18.0 fixed that at `src/instream/model_init.py:396`. Two consecutive full-suite runs after the fix produced **identical** failing output (`1 failed, 872 passed, 9 skipped`). The failure is now deterministic.

2. **It's not a sibling-test mutation.** All 7 sibling tests in `TestMarineLifecycleE2E` are read-only (verified in v0.18.0). They assert on counters and arrays without writing.

The leak is therefore upstream — some test runs before `test_marine_e2e.py` alphabetically and modifies state in a way that propagates into the `model` class-scoped fixture. Most likely candidates:
- **`test_calibration_ices.py`** (runs just before `test_marine_e2e.py` alphabetically) has its own class-scoped `model` fixture that runs a large 5-year simulation. If anything in `MarineDomain` or `InSTREAMModel.__init__` has a module-level cache, the calibration run's state leaks.
- **Numba JIT cache** — numba's `@njit` functions compile once per process and cache FP operations. If the calibration run warms up the cache with different inputs than the isolated marine_e2e run, subsequent compiled paths may produce slightly different FP results.
- **pytest's own `.pytest_cache/v/cache/lastfailed`** — unlikely but possible.

## Phase 3 — `spawn_defense_area` NetLogo semantic reconciliation

### Task 3.1: Write the failing test documenting both semantics

**Files:**
- Create: `tests/test_spawn_defense_semantics.py`

**Rationale**: NetLogo InSALMO 7.3 uses `trout-spawn-defense-area` at `InSALMO7.3_2023-07-06_ExampleA.nlogo:4200` as an **area in m²** subtracted from available spawning gravel per existing guarder (`set avail-area avail-area - (sum [trout-spawn-defense-area] of cell-guarders-here)`). The Python port at `src/instream/modules/spawning.py:186` uses it as a **Euclidean distance exclusion radius** (`valid_mask &= dist > defense_area`). These are fundamentally different: `200000` in NetLogo means "200,000 m² = 20 ha of gravel per guarder"; in Python it means "200 km exclusion radius".

The fix: rename the existing Python field to `defense_radius_cm` (preserving the distance semantic as a valid ecological model) and add a new `defense_area_m2` parameter matching NetLogo semantics. Both are optional and default to `0`.

**Cycle-1 scientific reviewer corrections**:

1. **Unit mismatch (CRITICAL)**: `CellState.area` is stored in **cm²** per the codebase convention. The plan initially passed `cell_spawn_areas = cs.area * cs.frac_spawn` (cm²) to be compared against `defense_area_m2` (m²). Off by a factor of 10,000. The v0.19.0 implementation must convert: `cell_spawn_areas_m2 = cs.area * cs.frac_spawn / 10000.0`.

2. **NetLogo divergence on binary-vs-continuous exclusion**: NetLogo subtracts a continuous amount from available gravel (so one guarder with small defense leaves room for others, partial-use); the Python implementation uses binary exclusion once cumulative defense exceeds cell area. Same end-state for well-defended cells but not bit-parity. **Accepted as a documented semantic equivalence, not strict NetLogo parity**. Phase 3's code comment must flag this.

3. **Multi-species shortcut (INFO)**: `guarders_here * defense_area_m2` assumes all guarders have the same `defense_area_m2`. NetLogo sums per-guarder values which could differ for multi-species. For v0.19.0 this is acceptable because `defense_area_m2` is a species-constant, and multi-species cells would be rare in freshwater reaches. Flag in a code comment.

4. **Pydantic alias (CRITICAL)**: the initial plan proposed THREE independent fields (`spawn_defense_radius_cm`, `spawn_defense_area_m2`, `spawn_defense_area` as alias). With `SpeciesConfig(extra="allow")` this gives three independent attributes and the alias enforcement is only in Python runtime fallback logic — the test `test_legacy_config_field_name_still_parses` accidentally validates the wrong code path. The v0.19.0 implementation must use Pydantic 2's `Field(validation_alias=AliasChoices(...))` to properly alias `spawn_defense_area` → `spawn_defense_radius_cm`.

## Phase 4 — Hard-scoped scite sweep on 6 species fields

**Rationale**: v0.18.0 introduced `configs/baltic_salmon_species.yaml` with 86 Chinook-copied fields. Retrieving Atlantic-salmon-specific literature for all 86 is out of scope for a single release. Phase 4 targets exactly 6 fields with the highest scientific leverage on calibration and ecological realism:

| Field | Why it matters |
|---|---|
| `spawn_fecund_mult` | Fecundity directly controls next-generation cohort size. **CRITICAL**: `create_redd` at `src/instream/modules/spawning.py` line ~236 uses `num_eggs = fecund_mult * weight^fecund_exp * egg_viability` — the formulation is **weight-based**, not length-based. Chinook values 690 × 0.552 give ~60,000 eggs for a 4 kg salmon female, an order of magnitude higher than Atlantic salmon literature. | `spawn_fecund_exp` | Weight-fecundity allometric exponent (NOT length-fecundity despite the name). Atlantic salmon value is much closer to 1.0 than 0.5. |

| `redd_devel_C` | Quadratic temperature coefficient |
